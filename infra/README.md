# Kluch — Azure deployment (Terraform)

Minimal, scale-to-zero Azure stack for the Kluch monorepo.

- **Region:** `italynorth` (Static Web App is global, pinned to `westeurope` metadata).
- **Subscription:** `cc97dbf2-f732-4fd1-b853-faa7d795df9e` (Azure Sponsorship).
- **Auth:** Terraform uses your `az login` session. No creds in the repo.

## What gets created

| Resource | Notes |
|---|---|
| `rg-kluch-prod` | Resource group (italynorth) |
| PostgreSQL Flexible Server | `B_Standard_B1ms`, v16, 32 GB, public access + "allow Azure services" firewall rule, DB `kluch` |
| Storage Account + `photos` container | Standard LRS, container access `blob` (public read of blobs) |
| Container Registry (ACR) | Basic, admin enabled |
| Log Analytics workspace | Required by the Container App environment |
| Container App Environment + `kluch-backend` | 0.25 vCPU / 0.5 GiB, min 0 / max 2 (scale-to-zero), external ingress on 8080 |
| Static Web App `kluch-console` | Free SKU |

Secrets (`pg_admin_password`, `session_secret`) are generated by `random_password` and
exposed as **sensitive** outputs. Read them with `terraform output -raw <name>`.

## Prerequisites

```sh
az account set --subscription cc97dbf2-f732-4fd1-b853-faa7d795df9e
cd infra
terraform init
```

## Deploy order

### 1. Stand up the infrastructure (placeholder backend image)

```sh
cd infra
terraform init
terraform apply
```

The Container App boots on the public placeholder image
(`mcr.microsoft.com/azuredocs/containerapps-helloworld:latest`) so the first apply
succeeds before we have a real image. This resolves the image chicken-and-egg: the
ACR exists after this step, ready to receive the backend image.

### 2. Build + push the backend image with ACR Tasks (no local Docker needed)

```sh
ACR=$(terraform output -raw acr_login_server)        # e.g. kluchacrxxxxxx.azurecr.io
ACR_NAME=${ACR%%.*}                                  # registry name without the domain
# Build context is the repo root (the Dockerfile is at ./Dockerfile).
az acr build -r "$ACR_NAME" -t kluch-backend:latest -f ../Dockerfile ..
```

### 3. Point the Container App at the real image

```sh
ACR=$(terraform output -raw acr_login_server)
terraform apply -var "backend_image=${ACR}/kluch-backend:latest"
```

This rolls a new revision. On boot the container runs migrate + seed + start
(see the repo `Dockerfile` CMD), so the database is provisioned on first run.

### 4. Build + deploy the Expo console to the Static Web App

```sh
# From the repo root:
export NVM_DIR="$HOME/.nvm"; . "$NVM_DIR/nvm.sh"; nvm use 22

BACKEND_URL=$(terraform -chdir=infra output -raw backend_url)
SWA_TOKEN=$(terraform -chdir=infra output -raw swa_api_token)

# IMPORTANT: always pass --clear. Metro caches the inlined EXPO_PUBLIC_* values, and a
# stale cache has twice baked in the localhost API URL (console then shows "Failed to fetch").
rm -rf apps/app/dist apps/app/.expo
EXPO_PUBLIC_API_URL="$BACKEND_URL" pnpm --filter @kluche/app exec \
  expo export --platform web --output-dir dist --clear

npx @azure/static-web-apps-cli deploy apps/app/dist \
  --deployment-token "$SWA_TOKEN" \
  --env production
```

Console default hostname: `terraform -chdir=infra output -raw swa_default_hostname`.

**Verify after deploy:** fetch the entry JS and confirm the real backend URL is baked in,
not `localhost:8080`:

```sh
JS=$(curl -s "https://$(terraform -chdir=infra output -raw swa_default_hostname)/" \
  | grep -oE "/_expo/static/js/web/[A-Za-z0-9._-]+\.js" | head -1)
curl -s "https://$(terraform -chdir=infra output -raw swa_default_hostname)${JS}" \
  | grep -oE "https://kluch-backend[^\"']*azurecontainerapps\.io|http://localhost:8080" | sort -u
```

### 5. Custom domain — `kluche.me` (phase 2)

Add these DNS records at the `kluche.me` registrar, then bind the custom domains in
Azure (Static Web App "Custom domains" blade and Container App "Custom domains" blade —
each shows the exact validation TXT it wants).

| Host | Type | Value |
|---|---|---|
| `app` (console) | CNAME | `<swa_default_hostname>` |
| `api` (backend) | CNAME | `<backend_fqdn>` |
| `*` (tenant sites, wildcard) | CNAME | `<backend_fqdn>` |

The Container App also requires a domain-validation TXT (`asuid.<host>` =
the app's custom-domain verification ID) before it will accept the binding;
Azure surfaces this value when you add the custom domain.

## Reading secrets / outputs

```sh
terraform output                       # non-sensitive
terraform output -raw backend_url
terraform output -raw acr_login_server
terraform output -raw postgres_fqdn
terraform output -raw storage_account_name
terraform output -raw swa_api_token        # sensitive
terraform output -raw pg_admin_password    # sensitive
terraform output -raw session_secret       # sensitive
```

## Validate without applying

```sh
terraform init
terraform validate
terraform plan -var 'pg_admin_password=Dummy_Pass_1234!' -out=tf.plan
rm -f tf.plan
```

## Notes / gotchas

- **State is local** (no remote backend configured). `*.tfstate*` is gitignored.
  State holds the generated DB password + session secret — do not commit it. Consider
  an `azurerm` backend (a storage container) before sharing this with a team.
- **PostgreSQL SSL:** `DATABASE_URL`/`DIRECT_DATABASE_URL` use `sslmode=require`.
  Flexible Server requires TLS; the app/driver must not disable it.
- **Storage env-switch:** the backend uses Azure Blob only when `AZURE_STORAGE_ACCOUNT`
  is set (Terraform sets it). Local Docker leaves it unset → LocalDiskStorage.
- **SWA region:** `kluch-console` lives in `westeurope` because SWA's allowed metadata
  regions exclude `italynorth`; the resource itself is globally distributed.
